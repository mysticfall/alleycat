# Character Physical Interaction API

> **Child spec of** [BODY-002: Character Physical Response System](../002-character-physical-response/index.md)

> **Status:** This spec defines the foundational polymorphic physical interaction API for source data providers,
> collision-target body-part interpretation, and generated body-part receipt notification. Production hand
> detection and broader reaction systems are future slices.

## Requirement

The system must provide a polymorphic physical interaction delivery mechanism where collision or overlap detection can
locate a collided receiver, and that receiver can interpret source-specific interaction data without coupling sensors to
reaction logic.

## Goal

Establish `AlleyCat.Interaction.Physical` contracts that separate pluggable interaction source data from receiver-owned
interpretation. Source components provide tagged interaction data, while collision or overlap paths identify whether the
collided object is associated with an `IPhysicalInteractionReceiver`. Receivers create suitable interaction payloads,
relay receipt notifications, and preserve generated body-part metadata. This slice implements the generic API,
generated body-part receiver proxies, and impact-only interaction data; hand detection and post-receive reactions remain
future work.

## User Requirements

1. Generated body parts from `DynamicPhysicalRig` must expose a standard receiver capability for external physical
   interaction sources.
2. Future hand probes, environment objects, and thermal zones must share a polymorphic source/receiver API without
   forcing impact-only fields onto every interaction type.
3. Collision or overlap detection must determine which receiver was contacted, while receivers determine whether and how
   to interpret the provided source data.
4. Receiver-owned bone metadata and optional tags must remain available so future systems can select or interpret the
   affected receiver without putting target fields into interaction payloads.
5. When a generated body part receives a supported interaction source, the owning rig must observe the receipt with the
   affected bone index and receiver tags so later systems can subscribe without polling body parts.
6. Interaction receipt must remain decoupled from reactions; receivers may return no interaction, and this slice must
   not perform IK reactions, combat, damage, haptics, or router dispatch.
7. Player XR hand authority must remain unchanged; future detection operates on hand motion data without modifying XR
   tracking input.

## Technical Requirements

1. Shared interaction API source files shall live under `game/src/Interaction/Physical/` and use the
   `AlleyCat.Interaction.Physical` namespace. Shared reusable metadata-tag contracts shall live under
   `game/src/Common/` and use `AlleyCat.Common`.
2. `AlleyCat.Common.ITagged` shall define reusable read-only tag metadata as `IReadOnlySet<string> Tags`.
3. `IPhysicalInteraction` shall be the common polymorphic interaction interface. It shall expose common source and
   world-space contact-point data only, and shall not use a kind enum.
4. `IImpactPhysicalInteraction` and its immutable implementation shall carry impact-specific velocity in addition to the
    common contact point. Impact is the only concrete interaction subtype in this slice.
5. `IPhysicalInteractionSource` shall compose `ITagged` as a marker capability for code that can initiate a
   physical interaction. It shall not own receiver polling, accept receiver tag filters, or carry event-specific contact
   data.
6. `IImpactPhysicalInteractionSource` shall expose source-owned impact state such as velocity. Event-specific contact
   points belong to the collision/receiver interaction path, not to the source capability.
7. `IPhysicalInteractionReceiver` shall compose `ITagged` and define `InteractWith`, a receiver-owned method that
   interprets a source, creates suitable interaction data, relays downstream notification, and returns the created
   interaction or `null`.
8. `PhysicalInteractionReceiveResult` is intentionally not part of the API. Rejection reasons, accepted magnitude,
   payload mutation, routing decisions, and reaction state are future concerns.
9. `PhysicalBodyPart3D` shall be a subclass of `AnimatableBody3D` implementing
   `IPhysicalInteractionReceiver`. It shall be generated by `DynamicPhysicalRig` and include runtime `BoneName`,
   `BoneIndex`, read-only `Tags`, exported `AuthoredTags`, `SourceShapeId`, and `OwningRig` metadata.
10. `PhysicalBodyPart3D` shall support impact sources by creating `ImpactPhysicalInteraction` from
     `IImpactPhysicalInteractionSource`, emitting one receipt signal, and returning the created interaction.
     Unsupported source types shall return `null` and emit nothing.
11. `PhysicalInteractionReceipt` shall live in `AlleyCat.Body` with the Godot signals that require it. It shall wrap
    `IPhysicalInteraction` because Godot C# signal parameters must be Variant-compatible and the domain interaction is
    a plain C# interface.
12. Body-part receipt signals shall include the receipt, `BoneIndex`, and a `string[]` snapshot of receiver tags. The
    body part shall not retain mutable delivery history such as last interaction or receive count.
13. `DynamicPhysicalRig` shall generate `PhysicalBodyPart3D` nodes, connect to each body-part receipt signal,
    and forward notifications with the receipt, bone index, tags, and receiving body-part proxy.
14. `DynamicPhysicalRig` remains responsible for generation, skeleton sync, proxy queries, and notification forwarding.
    It stays independent of IK target pipeline, interaction detection, combat, damage, haptics, routers, and reactions.
15. Runtime node casting shall be used to access receiver capability; exported interface references in scenes must be
    avoided.
16. Existing collision-processing components may implement interaction source capabilities directly when they own the
    initiation context. Additional source roles such as pressure can be added later without adding duplicate collision
    shapes or relay nodes. `PhysicalInteractionImpactSource3D` is intentionally not part of this slice; impact
    initiation is implemented directly by `HandDynamicBodyInteractionController`.
17. The mirror-room dynamic prop impact path shall reuse the existing right-hand IK target collision processing instead
    of adding a separate `Area3D` relay, extra hand collision shape, or child source node.
    `HandDynamicBodyInteractionController` shall initiate impact interactions directly from the IK target dynamic-body
    collision path and deliver itself to any `IPhysicalInteractionReceiver` found on the contacted body, its children,
    or its parent. Mirror-room dynamic props, including `test_ball.tscn` and `test_stick.tscn`, shall carry
    `RigidBodyImpactInteractionReceiver3D` and use the dynamic-interaction collision layer/group expected by the IK
    target query so they can create `ImpactPhysicalInteraction`, emit receipt notification, and apply an impulse to the
    referenced or parent `RigidBody3D`.

## In Scope

- Polymorphic physical interaction interfaces and impact-only initial subtype in `AlleyCat.Interaction.Physical`.
- Tagged source and receiver interfaces with read-only tag semantics.
- Tagged source marker contract for interaction initiators.
- Receiver-owned interaction creation and downstream relay contract.
- `PhysicalBodyPart3D` impact receiver implementation and signal emission.
- `DynamicPhysicalRig` generated receiver proxies and rig-level forwarding.
- Unit and integration tests for the API, absence of source-owned receiver queries, receiver creation, unsupported
  sources, signal snapshots, rig forwarding, generated proxy count, sync preservation, read-only tags, and the minimal
  real dynamic-prop impact contracts.
- Reusable right-hand IK target collision source initiation, plus mirror-room scenario validation through instancing and
  rigid-body impact receiver wiring on dynamic props including `test_ball.tscn` and `test_stick.tscn`.

## Out Of Scope

- Full production hand source implementation beyond the minimal IK target impact initiation path.
- `PhysicalInteractionImpactSource3D` or other helper source nodes that duplicate the IK target collision processor.
- Thermal zone or environment object source implementations.
- Post-receive reaction systems such as heat response, stagger, damage, haptics, audio, or visual cues.
- Consumer subscription and event-driven reaction pipeline beyond rig-level receipt forwarding and the minimal
  rigid-body impulse receiver used by mirror-room dynamic props.
- Final tuning of thresholds, filter constants, or anatomical limits.
- Networking or multiplayer synchronisation.

## Acceptance Criteria

1. User Requirements 1 and 4 validated: generated proxy bodies implement `IPhysicalInteractionReceiver` and expose
   receiver-owned bone metadata plus read-only tags.
2. User Requirements 2 and 3 validated: the API uses polymorphic interaction/source interfaces and receiver-owned
   interpretation without source-owned receiver polling, an interaction kind enum, or impact fields on the base
   interface.
3. User Requirement 5 validated: integration tests prove body-part receipts are forwarded by the owning rig with the
   receipt, bone index, tag snapshot, and receiving proxy.
4. User Requirements 6 and 7 validated: no reaction, IK, combat, haptic, router, or XR authority changes are included.
5. Technical Requirements 1-4 validated: shared API files are in the generic interaction namespace/folder, `ITagged`
   remains read-only, `IPhysicalInteraction` is polymorphic, and impact interactions carry event-specific contact data
   plus velocity through `IImpactPhysicalInteraction`.
6. Technical Requirements 5-6 validated: `IPhysicalInteractionSource` remains a tagged initiator marker without
   compatible receiver query methods, and `IImpactPhysicalInteractionSource` carries velocity but no contact point.
7. Technical Requirements 7-8 validated: receivers create and relay optional interactions, and no
   `PhysicalInteractionReceiveResult` contract exists.
8. Technical Requirements 9-12 validated: `PhysicalBodyPart3D` creates and emits impact interactions,
   returns `null` for unsupported source types, snapshots tags, and has no mutable receive history.
9. Technical Requirements 13-17 validated: `DynamicPhysicalRig` generates receiver proxies, forwards receipt signals,
   preserves sync behaviour, avoids unrelated systems, uses runtime receiver access, and the mirror-room dynamic prop
   impact path reuses the existing IK target collision processor without an `Area3D` relay or new hand collision shape.
10. User Requirements 2-3 and Technical Requirement 17 validated: mirror-room `test_ball.tscn` and `test_stick.tscn`
    instances are `RigidBody3D` dynamic props with the hand dynamic interaction group, collision layer and mask, plus a
    `RigidBodyImpactInteractionReceiver3D` child for receiver-owned impact interpretation.
11. Generated proxy count preserved: integration tests verify the same number of proxy bodies are generated as before
    the refactor.
12. Existing sync behaviour preserved: integration tests verify skeleton-to-proxy position/rotation sync continues to
    function.
